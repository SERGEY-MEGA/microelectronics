## Тема работы

Реализация сервера на Arduino Uno для вычисления контрольной суммы `CRC-8` сообщений, поступающих с ПК по интерфейсу `UART`, с выдачей результата по внешнему сигналу.

## Цель работы

Изучить принципы вычисления циклического избыточного кода `CRC-8`, освоить прием данных по последовательному интерфейсу `UART`, реализовать управление выдачей результата с помощью кнопки и сравнить два режима работы: со сбросом и без сброса накопленного остатка.

## Оборудование

- Arduino Uno
- тактовая кнопка
- соединительные провода
- персональный компьютер с Arduino IDE
- USB-кабель

## Схема подключения

- `D2 -> кнопка -> GND`
- `USB Arduino -> ПК`

Вход `D2` работает в режиме `INPUT_PULLUP`, поэтому в ненажатом состоянии на нем логическая единица, а при нажатии кнопка замыкает вход на землю.

## Краткое описание работы программы

Arduino работает как сервер, принимающий байты по интерфейсу `UART` со скоростью `9600 бод`. Каждый поступивший байт сразу включается в вычисление контрольной суммы `CRC-8`. Нажатие на кнопку, подключенную к выводу `D2`, инициирует вывод текущего значения CRC в монитор порта. В первой версии программы после вывода результата аккумулятор CRC сбрасывается, а во второй версии сохраняется, что позволяет продолжать вычисление для всех последующих сообщений без обнуления промежуточного остатка.

## Что такое CRC

`CRC` (`Cyclic Redundancy Check`) — это циклический избыточный код, применяемый для обнаружения ошибок при передаче или хранении данных. Смысл метода заключается в том, что исходное сообщение рассматривается как двоичный полином и делится по модулю `2` на заранее выбранный порождающий полином. Остаток от такого деления и является контрольной суммой.

CRC широко применяется:

- в последовательных протоколах передачи данных;
- в сетевых интерфейсах;
- в системах хранения данных;
- в промышленных и встроенных устройствах.

Главная цель CRC — быстро обнаружить случайные ошибки в передаваемых данных.

## Порождающий полином

В работе используется стандартный полином CRC-8:

`G(x) = x^8 + x^2 + x + 1`

В шестнадцатеричном виде в программе он задается как:

`0x07`

Старший член `x^8` в программной записи обычно не хранится явно, поэтому в коде остается только младшая часть полинома.

## Как выполняется вычисление CRC-8

Алгоритм реализован побитово:

1. В текущий остаток CRC включается очередной байт сообщения операцией `XOR`.
2. Далее выполняются восемь шагов обработки, по одному на каждый бит.
3. Если старший бит остатка равен `1`, остаток сдвигается влево и затем выполняется `XOR` с полиномом.
4. Если старший бит равен `0`, выполняется только сдвиг.
5. После обработки всех байтов получается итоговый остаток — контрольная сумма `CRC-8`.

В логическом смысле это эквивалентно делению двоичного полинома сообщения на порождающий полином по модулю `2`.

## Листинг программы

В репозитории подготовлены две версии:

- `lab3_crc8_reset/lab3_crc8_reset.ino` — версия со сбросом CRC после выдачи;
- `lab3_crc8_accumulate/lab3_crc8_accumulate.ino` — версия без сброса CRC после выдачи.

## Проверка базовой версии

Для проверки нужно открыть `Serial Monitor` со скоростью `9600 бод` и выбрать режим `No line ending`.

### Эксперимент 1. Передача одного символа `A`

Передается один байт:

`0x41`

После нажатия кнопки программа выдает:

`CRC-8 = 0xC0`

### Эксперимент 2. Передача строки `AAA`

После сброса аккумулятора передаются три байта:

`0x41 0x41 0x41`

После нажатия кнопки программа выдает:

`CRC-8 = 0x63`

## Почему при передаче `AAA` CRC меняется

Контрольная сумма CRC зависит не только от того, какие символы передаются, но и от длины сообщения и порядка следования байтов. После обработки первого символа формируется промежуточный остаток. При поступлении второго и третьего символов именно этот остаток участвует в дальнейших сдвигах и операциях `XOR`, поэтому конечный результат для `AAA` отличается от результата для одного символа `A`.

Иными словами, `CRC(A)` и `CRC(AAA)` — это контрольные суммы для двух разных сообщений, поэтому совпадать они не обязаны.

## Проверка измененной версии

Во второй версии после нажатия кнопки аккумулятор CRC не сбрасывается.

Если:

1. отправить `A`;
2. нажать кнопку;
3. затем отправить `AAA`;
4. снова нажать кнопку,

то будет рассчитана контрольная сумма уже не для новой строки `AAA`, а для общей последовательности `AAAA`.

В этом случае ожидаемый результат:

`CRC-8 = 0xEE`

## Итоги сравнения двух версий

### Версия со сбросом

- удобна для независимой проверки отдельных сообщений;
- после каждого нажатия кнопки начинается новое вычисление CRC.

### Версия без сброса

- сохраняет предыдущее состояние аккумулятора;
- позволяет накапливать CRC для непрерывного потока данных;
- лучше соответствует режиму непрерывной обработки последовательности сообщений.
